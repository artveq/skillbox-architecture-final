# 5. Описание рисков реализации (бизнес-риски и технические риски)

## 1. Бизнес-риски

### 1.1. Низкая вовлечённость пользователей
**Описание:** Пользователи могут не использовать социальные функции, рекомендации или геймификацию.  
**Последствия:** снижение DAU/MAU, падение удержания, низкая ценность приложения.  
**Причины:** слабая мотивация, сложный интерфейс, отсутствие друзей в системе.  
**Меры по снижению:**
- персональные рекомендации и уведомления,
- социальные механики (группы, челленджи),
- A/B-тестирование UX,
- адаптация под разные уровни подготовки.

### 1.2. Недостаточная монетизация
**Описание:** Пользователи могут игнорировать промо и рекомендации по инвентарю.  
**Последствия:** низкий ROI, снижение интереса бизнеса к развитию продукта.  
**Причины:** нерелевантные предложения, слабая интеграция с e-commerce.  
**Меры по снижению:**
- ML-персонализация промо,
- рекомендации на основе износа инвентаря,
- интеграция с программами лояльности.


### 1.3. Конкуренция с крупными фитнес-платформами
**Описание:** Рынок насыщен (Strava, Nike Run Club, Garmin Connect).  
**Последствия:** сложность привлечения пользователей.  
**Причины:** отсутствие уникальных фич, слабая дифференциация.  
**Меры по снижению:**
- упор на социальные группы и локальные активности,
- глубокая интеграция с товарами бренда,
- уникальные челленджи и региональные события.


### 1.4. Репутационные риски
**Описание:** Ошибки в данных, некорректные рекомендации или токсичность в группах могут повредить бренду.  
**Последствия:** негативные отзывы, снижение доверия.  
**Причины:** слабая модерация, ошибки аналитики.  
**Меры по снижению:**
- инструменты модерации,
- валидация данных,
- прозрачные правила поведения.


### 1.5. Несоответствие локальным законам (GDPR, локальные регуляции)
**Описание:** Нарушение требований по хранению и обработке данных.  
**Последствия:** штрафы, запрет на работу в регионе.  
**Причины:** неправильная геолокация данных, отсутствие согласий.  
**Меры по снижению:**
- региональное хранение данных,
- юридическая экспертиза,
- прозрачная политика конфиденциальности.

## 2. Технические риски

### 2.1. Пиковые нагрузки при массовых соревнованиях
**Описание:** Одновременные тренировки десятков тысяч пользователей.  
**Последствия:** задержки, деградация сервисов, потеря данных.  
**Причины:** синхронные вызовы, недостаточная масштабируемость.  
**Меры по снижению:**
- event-driven архитектура,
- autoscaling,
- очереди и буферизация телеметрии,
- нагрузочное тестирование.

### 2.2. Интеграция слишком большого количества устройств
**Описание:** Разные протоколы, BLE-устройства, нестабильные каналы.  
**Последствия:** потеря данных, неконсистентность.  
**Причины:** отсутствие стандартов, низкое качество связи.  
**Меры по снижению:**
- IoT Gateway с буферизацией,
- стандартизированные SDK,
- QoS уровни доставки.

### 2.3. Слишком высокие требования к безопасности и приватности
**Описание:** Обработка чувствительных данных (локация, здоровье).  
**Последствия:** утечки, штрафы, потеря доверия.  
**Причины:** ошибки конфигурации, слабая сегментация.  
**Меры по снижению:**
- Zero Trust,
- шифрование данных,
- сегментация API (External / Internal / IoT),
- аудит и логирование.

### 2.4. Проблемы геораспределённых данных и задержки
**Описание:** У системы будут пользователи по всему миру, а значит и разные регионы хранения.  
**Последствия:** задержки, сложность репликации.  
**Причины:** распределённые базы, сетевые ограничения.  
**Меры по снижению:**
- Data Locality,
- кэширование,
- CDN,
- минимизация межрегиональных запросов.

### 2.5. Сложность мультиоблачной архитектуры
**Описание:** Разные облака означают разные инструменты, IAM, сети.  
**Последствия:** рост стоимости DevOps, сложность настройки и мониторинга.  
**Причины:** отсутствие единого провайдера, специфика системы  
**Меры по снижению:**
- стандартизация CI/CD,
- единый слой логирования и мониторинга,
- использование контейнеризации и оркестрации.
